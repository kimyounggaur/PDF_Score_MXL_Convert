# Audiveris (오픈소스 OMR) — 검증된 사실 정리 (2026-06 기준)

## 최신 버전 / 릴리스 형태 / Java 요구
- **최신 안정 버전: 5.10.2 (2026-03-27 릴리스, "Bug fix for biased notes")**. 질문의 "5.4/5.5" 전제는 구버전이며, 현재는 5.10.x 대까지 진행됨. (출처: github.com/Audiveris/audiveris/releases, README)
- **릴리스 형태**: 소스 빌드 + **각 OS용 설치 프로그램(installer)** 둘 다. **5.5 버전부터** Windows/macOS/Linux 설치 프로그램이 제공되며 **JRE가 번들로 포함**됨 — "The application comes with its own Java Runtime Environment (JRE). Therefore, there is no need for the user to install a specific JRE." 형태: Windows MSI, macOS DMG, Linux DEB, Flathub Flatpak. (출처: install/binaries 핸드북)
- **Java 버전(번들/빌드)**: 최근 버전이 **Java 24 + Gradle 9** (5.7.0)로 업그레이드되었고, **5.8.1은 Java 25 + Gradle 9.1**로 추가 업그레이드. 설치 프로그램에 적합한 JRE가 동봉됨. (출처: releases 페이지)
- **빌드 도구**: git, gradle, JDK 필요. 구 5.1 문서는 JDK 7/8을 언급하나 이는 **불확실/구버전 정보**이며 현행은 위 Java 24/25 기준. (출처: 5.1 sources 문서 = 구버전)

## Batch / Headless CLI (실제 존재하는 플래그만 — 현행 핸드북 기준)
구문: `audiveris [OPTIONS] [--] [INPUT_FILES]`. 검증된 표준 플래그:
- **`-batch`** — GUI 없이 실행 (인자 없음)
- **`-transcribe`** — book 전체를 transcribe (인자 없음)
- **`-export`** — MusicXML(.mxl) export. **`-export`는 `-transcribe`를 암시적으로 포함**함 (인자 없음)
- **`-print`** — book을 PDF로 출력 (인자 없음)
- **`-output <DIRNAME>`** — 출력 기본 폴더 지정 (.omr/.mxl 등 저장 위치)
- **`-sheets N M X-Y`** — 처리할 sheet 번호/범위 선택 (예: `1 4-5`)
- **`-step <STEPNAME>`** — 목표 처리 단계 지정 (LOAD, BINARY, SCALE, GRID, HEADERS, …, PAGE 등)
- **`-force`** — 목표 단계 이미 도달했어도 재처리 (목표 step 또는 `-transcribe` 있을 때만 유효)
- **`-save`** — batch 모드에서 각 단계 성공 시 .omr 저장
- **`-swap`** — 처리 후 각 sheet를 메모리에서 swap out (대용량 시 메모리 절약)
- **`-constant KEY=VALUE`** — 애플리케이션 constant(인식 파라미터) 설정. **`-option`은 구 명칭이며 둘 다 지원됨** ("'-constant' is a better name than the old '-option', but both names are supported"). KEY는 fully-qualified 이름.
- 기타: `-help`, `-version`, `-playlist FILE.XML`, `-upgrade`, `-run CLASS`, `-sample`, `-annotate`, `--`(이후는 파일 경로)
- 예시(검색 결과 인용): `audiveris -batch -print -transcribe -export -output "audivers" "test.png"`
- **입력으로 PDF/이미지 직접 가능**: 예. 입력은 .omr 또는 이미지 파일. 지원 포맷: **PDF, TIFF, JPG, PNG, BMP**. PDF/TIFF는 멀티 이미지(멀티 페이지) 지원, PNG/JPEG/BMP는 단일 이미지. (출처: cli 핸드북, Main-Concepts wiki)

## 출력 디렉터리 / 파일명 규칙 / .mxl 다중 산출
- **radix 기반 서브폴더**: 입력 파일명(radix)으로 전용 폴더 생성. 예) `Bach_Fugue.tif` → `Bach_Fugue/` 폴더 안에 `Bach_Fugue.omr`, `Bach_Fugue.mxl`, (선택) `Bach_Fugue-samples.zip`. (출처: standard folders 5.1 문서)
- **.omr** = Audiveris 프로젝트 파일(XML 기반 OMR 데이터), **.mxl** = 압축 MusicXML. `-output`으로 기본 출력 폴더 변경.
- **.mxl가 여러 개 나오는 조건**:
  - **Book export 기본 동작**: book 내에서 발견된 **각 movement마다 별도의 .mxl 1개** 생성 → 멀티 movement이면 .mxl 여러 개.
  - `useOpus` 옵션을 켜면 separate movement들을 담은 **global opus 단일 .mxl** 1개로 export. 상수명: **`org.audiveris.omr.sheet.BookManager.useOpus`** (기본 false). 단, opus는 MusicXML 정의이나 지원 에디터가 거의 없음.
  - **Sheet export**: sheet 내 각 page마다 별도 .mxl 1개.
  - 멀티 book/멀티 입력 파일은 각 입력별 radix 폴더로 분리됨. (출처: outputs/mxl 5.1 문서)

## 정확도 영향 설정
- **해상도/interline**: **300 DPI**가 A4/US Letter 표준에서 적정, 작은 심볼은 **400 DPI**. 더 좋은 기준은 **두 staff line 사이 거리(interline)가 약 20픽셀**이 되도록 스캔. 하한 200 DPI 미만은 세부 손실, 상한 500 DPI 초과는 CPU/메모리 낭비. **흑백/컬러보다 grayscale 권장**(Audiveris가 자체 adaptive 알고리즘으로 binarize). (출처: scanning 핸드북)
- **OCR 언어 설정**:
  - 기본 언어 상수: **`org.audiveris.omr.text.Language.defaultSpecification`** (초기값 `eng`). batch에서: `-constant org.audiveris.omr.text.Language.defaultSpecification=ita+eng` 처럼 plus-separated 코드 리스트(`fra+eng+ita`). 언어 너무 많으면 인식 느려짐.
  - **TESSDATA_PREFIX**: 정의되어 디렉터리를 가리키면 그 위치 사용, 없으면 user config 디렉터리에 `tessdata` 폴더 자동 생성. **설정은 필수 아님**(기존 구성 재사용 목적). 언어 추가는 `Tools → Languages` 메뉴(인터넷 필요).
  - **Tesseract 버전**: 현행 languages 페이지는 **4.x 언어 파일** 다운로드를 명시. README/최근 빌드는 텍스트 인식을 **Tesseract OCR 5.5.1 (javacpp 바인딩)** 에 위임한다는 보고도 있음 → 현행은 **Tesseract 4.x/5.x 라인** (구 "3.04 필요"는 옛 정보). 별도 Tesseract 실행파일 설치 불필요(라이브러리로 사용). **버전 일관성은 다소 불확실** — 정확히는 build.gradle/현행 핸드북으로 재확인 권장.
- **`-constant`로 바꿀 수 있는 인식 파라미터 예**: 위 `Language.defaultSpecification`, `BookManager.useOpus` 등 fully-qualified constant. GUI `Tools → Constants`와 동일. (출처: cli, languages 핸드북)

## Docker / Headless 운영 함정
- **OCR 의존성**: Tesseract는 빌드/실행 모두 **필수**. 라이브러리(javacpp/JNA 계열)로 호출되므로 실행파일은 불필요하나, **언어 데이터 파일(tessdata)과 TESSDATA_PREFIX 경로 설정이 핵심**. Linux 빌드 시 "No OCR is available" 오류 다수 보고. (출처: 깃허브 issue #698, #675, #628; Environment wiki)
- **PDF 입력**: PDF 래스터화 의존(ghostscript/poppler류) 가능성 — 핸드북에 **명시적 poppler/ghostscript 의존 문구는 직접 확인 못 함 → 불확실**. PDF 직접 입력은 지원됨.
- **java.awt headless / fontconfig**: Audiveris의 `-batch`는 GUI 없이 동작하나, **`-Djava.awt.headless=true`, fontconfig(폰트 설정) 의존**은 공식 문서에서 직접 명시한 구절을 찾지 못함 → **불확실**(일반적 Java GUI 라이브러리/심볼 폰트 의존상 컨테이너에서 fontconfig·freefont 누락 시 문제 가능하나 공식 근거 미확인). 심볼 렌더링용 **FreeFont(MusicalSymbols 등) 폰트** 필요 가능성 있으나 공식 확인 불확실.

# Claude API 악보 이미지 분석/대조 — 검증된 사실 정리

## 1. Messages API 이미지 입력

- **지원 포맷**: JPEG, PNG, GIF, WebP (`image/jpeg`, `image/png`, `image/gif`, `image/webp`)만 지원. 애니메이션은 미지원이며 **GIF/애니메이션은 첫 프레임만 사용**된다. 출처: https://platform.claude.com/docs/en/build-with-claude/vision.md
- **입력 방식 3가지**: ① base64 인라인, ② 공개 URL(`source.type: "url"`), ③ Files API 업로드 후 `file_id` 참조(beta `files-api-2025-04-14`). 멀티턴/에이전트 워크플로(여러 페이지 반복 대조)에서는 매 요청마다 전체 히스토리가 재전송되므로 **base64보다 Files API의 `file_id` 참조가 페이로드/지연 면에서 유리**. 출처: 같은 vision 문서.
- **한 요청당 이미지 개수 상한**:
  - API: **600개/요청** (일반 모델), **100개/요청** (200k 컨텍스트 모델). claude.ai: 20개/턴.
  - 단, 한 요청에 **이미지 20개 초과** 시 더 엄격한 per-image 치수 제한이 적용됨 — 초과분은 `invalid_request_error`("many-image requests"). 모든 플랫폼에서 안전하려면 각 이미지를 **양 변 2000px 이하**로 리사이즈하거나 이미지+문서 블록을 20개 이하로 유지. 출처: vision 문서 Request limits.
- **이미지 크기/치수 상한**: 최대 치수 **8000×8000 px**. 파일 크기 **10MB (base64, Claude API 직접)** / 5MB (Bedrock·Vertex) / 10MB (claude.ai). 요청 전체 크기 상한 **32MB**(표준 엔드포인트)에 먼저 걸릴 수 있음.
- **권장/최대 해상도와 초과 시 동작** (모델 티어별 — 중요):
  | 티어 | 모델 | 최대 long edge | 최대 visual token |
  |---|---|---|---|
  | High-resolution | Fable 5, Mythos 5, **Opus 4.8**, Opus 4.7 | **2576 px** | 4784 |
  | Standard | 그 외 전체 (**Sonnet 4.6** 포함) | **1568 px** | 1568 |
  - 두 한도(long edge 또는 visual token) 중 하나라도 초과하면 처리 전 **자동 다운스케일**(거부가 아님). 고해상도 지원은 해당 모델에서 자동, 베타 헤더/옵트인 불필요.
  - **악보 대조 관점 주의**: `claude-sonnet-4-6`은 standard 티어이므로 long edge 1568px로 다운스케일됨 → 빽빽한 악보에서 텍스트/음표 가독성 손실 위험. 고해상도가 필요하면 `claude-opus-4-8`(2576px)을 쓰거나 페이지를 잘라서(crop) 보내는 것이 정확도에 유리. 출처: vision 문서 Resolution and token cost.
- **이미지 토큰 비용 추정식** (현재 공식 — 과거 `width×height/750` 휴리스틱은 폐기됨):
  - Claude는 이미지를 **28×28 px 패치(=1 visual token)** 단위로 처리. **`visual tokens = ⌈width/28⌉ × ⌈height/28⌉`** (다운스케일 후 치수 기준).
  - 예시(공식): 1000×1000 → 1296 tokens; 1092×1092 → 1521; 1920×1080 → standard 1560(다운스케일됨)/high-res 2691; 2000×1500 → standard 1564/high-res 3888; 3840×2160 → standard 1560/high-res 4784(캡).
  - 비용 = visual tokens × 모델 input 단가. 예: Opus 4.8($5/1M)에서 1000×1000 이미지 ≈ $6.48/1000장, 4K ≈ $23.92/1000장. 출처: vision 문서.

## 2. 구조화 출력 보장 — tool use vs system 프롬프트

- **가장 견고한 방법은 "강제된 JSON 스키마"**이며, 두 가지 모두 스키마 검증을 제공한다 (둘 다 system-프롬프트로 "JSON만 출력" 지시하는 방식보다 견고):
  - **Structured Outputs** (`output_config: {format: {type: "json_schema", schema: ...}}`): 응답 형식을 스키마에 맞게 **제약**. SDK의 `client.messages.parse()`(Python Pydantic / TS Zod) 사용 권장 — 자동 검증. 지원 모델: Fable 5, **Opus 4.8**, **Sonnet 4.6**, Haiku 4.5.
  - **Strict tool use** (`strict: true`를 tool 정의에 — `tool_choice`가 아님): `tool_use.input`이 스키마에 정확히 검증됨을 보장. 스키마에 `additionalProperties: false` + `required` 필수.
- **권장(악보 대조 결과를 JSON으로 받을 때)**: `output_config.format`로 출력 스키마를 강제하는 쪽이 단순하고 견고. tool use(strict)는 "Claude가 도구를 호출해야만 한다"는 흐름이 자연스러울 때. **system 프롬프트로 "JSON-only" 강제는 모델이 어기거나 preamble을 붙일 수 있어 가장 약함** — 채택하지 말 것.
- **주의/제약**: 새 스키마는 첫 요청에 1회 컴파일 비용(24h 캐시). `stop_reason == "refusal"`이면 스키마와 불일치할 수 있음. `output_config.format`은 **Citations와 동시 사용 불가(400)**, 그리고 메시지 prefill과도 비호환. 출처: claude-api 스킬 `shared/tool-use-concepts.md` (Structured Outputs), python/TS `tool-use.md`.

## 3. Prompt Caching

- **무엇을 캐시하면 좋은가**: 변하지 않는 **시스템 프롬프트(채점/대조 규칙)**, **출력 JSON 스키마**, **공통 지시/few-shot 예시** 같은 안정적 접두부(prefix). 렌더 순서는 `tools → system → messages`이며 **접두 일치(prefix match)** — 접두부에서 1바이트라도 바뀌면 그 뒤 전부 무효화.
- **악보별 변동 부분(개별 페이지 이미지·페이지 번호·질문)은 마지막 cache_control 브레이크포인트 뒤에 배치**. 페이지마다 다른 이미지는 캐시되지 않지만, 공통 시스템/스키마는 페이지 전체에서 재사용 가능.
- **비용 절감 효과**: 캐시 읽기 ≈ 기본 input 단가의 **0.1×** (약 90% 절감). 캐시 쓰기 = **1.25×**(5분 TTL) / **2×**(1시간 TTL). 손익분기: 5분 TTL은 2회 요청, 1시간 TTL은 3회 이상에서 이득.
- **사용법 개요**: `cache_control: {type: "ephemeral"}`를 안정 접두부 마지막 블록에 둠(또는 요청 최상위 `cache_control`로 자동 배치). 요청당 **최대 4개 브레이크포인트**. **최소 캐시 가능 길이는 모델별**: Opus 4.8/4.7 = 4096 토큰, Fable 5/Sonnet 4.6 = 2048 토큰 (그 미만이면 조용히 캐시 안 됨, 오류 없음). 검증은 `usage.cache_read_input_tokens`로 — 반복 요청에서 0이면 무효화 요인(시스템 프롬프트 내 `datetime.now()`, 비결정적 JSON 직렬화, 가변 tool 셋) 의심. 출처: 스킬 `shared/prompt-caching.md`.

## 4. Message Batches API (다수 페이지 비동기 병렬)

- **가능**. `POST /v1/messages/batches`로 다수 페이지를 비동기 일괄 처리. 비전·tool·caching 등 **모든 Messages API 기능 지원**.
- **비용/지연 특성**:
  - **표준 단가의 50% 할인** (모든 토큰 사용량).
  - 배치당 최대 **100,000 요청 또는 256MB**. 대부분 **1시간 내** 완료, 최대 24시간. 결과는 생성 후 29일간 보관.
  - **결과는 순서 무보장** — 반드시 `custom_id`로 매칭(위치 인덱스 X).
- **악보 대조 적합성**: 실시간이 아닌 "여러 페이지를 한꺼번에 채점/대조"에 비용·처리량 면에서 적합. 다만 latency가 민감하면(즉답 필요) 동기 Messages API 사용. 출처: 스킬 `python/claude-api/batches.md`. **가용성 주의**: Batches는 1P Claude API에서만(✅), **Bedrock·Vertex·Foundry 미지원** — `shared/platform-availability.md`.

# MusicXML / MXL 포맷 및 검증 정리

모든 항목은 W3C MusicXML 4.0 공식 명세(구 MakeMusic/Recordare 자료를 W3C가 호스팅)에 근거. 환각 없이 출처 명시.

## 1. .mxl(압축) 내부 구조 — zip 레이아웃

- **컨테이너 = ZIP**: .mxl은 DEFLATE(RFC 1951)로 압축된 ZIP 아카이브. 내부에 비압축 MusicXML(`.xml`/`.musicxml`)과 선택적 대체본(PDF, 오디오)을 담음. (출처: https://www.w3.org/2021/06/musicxml40/tutorial/compressed-mxl-files/)
- **첫 엔트리 `mimetype` 관례 (EPUB과 동일 규약)**:
  - ZIP의 **첫 번째 파일**이어야 하며 이름은 정확히 `mimetype`.
  - 내용은 정확히 `application/vnd.recordare.musicxml` (개행/패딩 없이).
  - **반드시 STORED(비압축)·비암호화**, **zip 로컬 헤더에 extra field 없음**, **US-ASCII**, **선행 공백/BOM 금지**. 앱이 파일 앞부분 바이트만 읽어 압축 MusicXML임을 식별하기 위함. (출처: https://www.w3.org/2021/06/musicxml40/tutorial/compressed-mxl-files/)
- **`META-INF/container.xml`**: 시작점(rootfile)을 지정. `<container>` → `<rootfiles>` → `<rootfile full-path="..." media-type="..."/>`.
  - **MusicXML 루트는 첫 번째 `<rootfile>`에 기술**해야 함. 그 문서요소는 `<score-partwise>`, `<score-timewise>`, 또는 `<opus>`.
  - `full-path`는 아카이브 루트 기준 상대경로(예: 점수 파일은 보통 루트 또는 하위 폴더에 위치). `container.xml`은 `container.dtd`로 정의되며 별도 `.xsd`는 없음. (출처: https://www.w3.org/2021/06/musicxml40/container-reference/elements/container/ , https://www.w3.org/2021/06/musicxml40/listings/container.xsd/)
- **미디어 타입**: 압축본 `application/vnd.recordare.musicxml`, 비압축 XML `application/vnd.recordare.musicxml+xml`. (출처: https://www.w3.org/2021/06/musicxml40/tutorial/compressed-mxl-files/)
- **재압축 시 주의 (실무 핵심)**:
  - 일반 zip 라이브러리로 단순 재압축하면 `mimetype`이 첫 엔트리가 아니거나 DEFLATE로 압축되어 규약 위반 가능 → **`mimetype`을 STORED·extra-field 없이 첫 엔트리로 명시 삽입** 후 나머지를 추가하는 순서를 강제해야 함.
  - 다만 대다수 리더(MuseScore, Finale 등)는 mimetype 규약이 느슨해도 `META-INF/container.xml`만 정확하면 읽는 경우가 많음. 그러나 **명세 준수 = 호환성 보장**이므로 엔트리 순서/STORED 플래그를 직접 제어하라(예: Node의 `yazl`로 `mimetype`은 `compress:false`, Python `zipfile`은 `ZIP_STORED`). (근거: 위 명세의 mimetype 제약)

## 2. score-partwise vs score-timewise + 핵심 요소

- **partwise**: `score-header` 뒤에 `<part>`들이 오고 각 part 안에 `<measure>`들 (파트 우선, 실무 표준). **timewise**: `<measure>`들이 오고 각 measure 안에 `<part>`들 (시간 우선, 드묾). 두 표현은 XSLT로 상호 변환 가능. (출처: https://usermanuals.musicxml.com/MusicXML/Content/EL-MusicXML-score-partwise.htm , https://usermanuals.musicxml.com/MusicXML/Content/EL-MusicXML-score-timewise.htm)
- **measure(partwise)의 자식 순서(선택·반복, music-data 그룹)**: `note`, `backup`, `forward`, `direction`, `attributes`, `harmony`, `figured-bass`, `print`, `sound`, `listening`, `barline`, `grouping`, `link`, `bookmark` — 이들은 **시간 순서대로 자유 배치**(고정 순서 아님, 다만 시간 위치를 backup/forward로 제어). `<measure number="1">`의 number 속성 필수. (출처: https://www.w3.org/2021/06/musicxml40/musicxml-reference/elements/measure-partwise/)
- **`<note>` 자식 순서(엄격 고정)**: `(grace|cue)? → (pitch|unpitched|rest) → duration → tie(0–2) → instrument* → footnote? → level? → voice? → type? → dot* → accidental? → time-modification? → stem? → notehead? → notehead-text? → staff? → beam(0–8) → notations* → **lyric*** → play? → listen?`. `<pitch>`는 `<step>`(A–G) → `<alter>`(반음, ±1=#/♭, ±0.5 미분음) → `<octave>` 순. **lyric은 거의 끝쪽**(notations 뒤). (출처: https://www.w3.org/2021/06/musicxml40/musicxml-reference/elements/note/)
- **`<harmony>`(코드기호)**: measure 내 music-data 요소. 자식 순서 `(root|numeral) → kind(필수) → inversion? → bass? → degree*`. root는 `<root-step>`+`<root-alter>`. **관례상 적용 대상 음표 바로 앞에 배치**하며, 정확한 가로 위치는 `<offset>`(divisions 단위) 및 `default-x/relative-x`로 조정. (출처: https://www.w3.org/2021/06/musicxml40/musicxml-reference/elements/harmony/)
- **`<lyric>`**: `<text>`(+`<syllabic>`, `<elision>`, `<extend>`)로 가사 음절. `number`/`name`으로 절 구분. (출처: https://www.w3.org/2021/06/musicxml40/musicxml-reference/elements/lyric/)
- **`<attributes><time>`**: `<beats>`/`<beat-type>`로 박자표(예 4/4). divisions(분할능)도 attributes에. (출처: https://www.w3.org/2021/06/musicxml40/musicxml-reference/elements/measure-partwise/)
- **`<print new-page="yes" new-system="yes">`**: yes/no로 페이지/시스템(단) 강제 분할. 보통 measure의 첫 음악데이터에 적용. `page-number`는 `new-page="yes"`일 때만. 자식 `<page-layout>/<system-layout>/<staff-layout>`은 현재 페이지/시스템/보표에만 적용, 이후는 defaults로 복귀. (출처: https://www.w3.org/2021/06/musicxml40/musicxml-reference/elements/print/)

## 3. 좌표 정보 (tenths) → 픽셀 역매핑

- **단위 = tenths(인터라인 간격의 1/10)**: 모든 거리(마진, 페이지 크기, default-x/y)는 tenths로 통일. tenths→mm 변환은 `<defaults><scaling>`의 `<millimeters>`/`<tenths>` 비율로 결정. (출처: https://www.w3.org/2021/06/musicxml40/musicxml-reference/elements/scaling/ , https://usermanuals.musicxml.com/MusicXML/Content/EL-MusicXML-scaling.htm)
- **default-x/default-y 의미(컨텍스트 의존)**: default-x=0은 음표/현재 음악 위치의 좌측 끝, default-y=0은 현재 보표의 위쪽 기준. direction 등은 default-x를 **현재 measure 시작점 기준**, default-y는 보표 위 barline 기준으로 측정. (출처: https://usermanuals.musicxml.com/MusicXML/Content/EL-MusicXML-defaults.htm , https://www.oxygenxml.com/samples/xml-schema-documentation/MusicXML-Schema/musicxml31.html)
- **`<system-layout>`/`<page-layout>`**: defaults 안에 있으면 전체 기본값, `<print>` 안에 있으면 해당 시스템/페이지에만 적용. 시스템 간 거리, 마진 등을 tenths로 지정. (출처: https://www.w3.org/2021/06/musicxml40/musicxml-reference/elements/system-layout/)
- **measure를 픽셀 영역으로 역매핑할 때 고려사항**:
  - MusicXML은 **measure 단위의 폭/픽셀 박스를 직접 제공하지 않음**. `<print new-system/new-page>`로 어느 measure에서 줄/페이지가 바뀌는지 추론하고, measure별 `width` 속성(tenths)을 누적해 가로 위치를 계산해야 함.
  - 절대 픽셀화하려면 **scaling(tenths↔mm) + DPI**를 곱해야 하며, default-x/y는 컨텍스트 기준점이 제각각이라 누적 좌표계를 직접 구성해야 함.
  - 많은 파일이 default-x/y·width를 생략(렌더러가 자동 배치) → 이 경우 좌표 역매핑 불가능, **렌더러(OSMD/Verovio)의 출력 SVG 좌표를 쓰는 편이 견고**. PDF→MXL 변환물은 OMR 추정 좌표라 신뢰도 편차 큼.

## 4. 검증 도구 / 방법

- **XSD 위치/배포 (권장 1순위)**: MusicXML 4.0 XSD는 DTD보다 엄격. W3C GitHub(`w3c/musicxml`)에서 XSD+DTD를 zip으로 일괄 배포. 핵심 스키마 `musicxml.xsd`. 검증 URI는 `http://www.musicxml.org/xsd/musicxml.xsd`. **DTD는 4.0부터 deprecated**. (출처: https://www.musicxml.com/for-developers/musicxml-xsd/ , https://www.w3.org/2021/06/musicxml40/listings/musicxml.xsd/)
- **xmllint --schema (실무 권장)**:
  - `xmllint --schema musicxml.xsd score.musicxml --noout`
  - **주의**: 배포 XSD의 `xml.xsd` import가 외부 URL(`http://www.musicxml.org/xsd/xml.xsd`)을 가리켜 네트워크 의존·매우 느림. **schemaLocation을 로컬 `xml.xsd`로 패치**하면 ~15초 → ~0.04초로 단축. 오프라인은 `--nonet` 필수. (출처: https://blog.karimratib.me/2020/11/17/validate-musicxml.html)
- **jing(RelaxNG)/trang**: MusicXML 공식 배포는 XSD/DTD라 RelaxNG는 trang으로 변환 후 jing 사용 가능하나 비표준 경로 → 일반적으로 XSD 직행이 나음. (실무상 잘 쓰지 않음)
- **music21**: `converter.parse()`가 파싱 시 구조 오류에 예외를 던지는 **파서 기반 검증**(엄밀한 XSD 검증 아님). 시맨틱 점검·라운드트립(파싱→재내보내기)에 유용. (출처: https://music21.org/music21docs/moduleReference/moduleConverter.html , https://github.com/cuthbertLab/music21/blob/master/music21/musicxml/m21ToXml.py)
- **가장 견고한 실무 조합**: **(1) XSD 구조 검증(xmllint, 로컬 xml.xsd 패치) + (2) 시맨틱/XPath 또는 music21 파싱 검증**의 2단계. XSD만으로는 박자 합 같은 음악적 정합성을 못 잡음. (출처: https://blog.karimratib.me/2020/11/17/validate-musicxml.html)

## 5. raw XML 편집(fast-xml-parser) 시 깨지기 쉬운 지점

- **요소 순서 규칙**: `<note>` 자식은 **고정 순서**(§2). fast-xml-parser는 키 객체로 파싱하므로 재직렬화 시 순서가 뒤섞이면 XSD 위반. `preserveOrder: true` 옵션으로 순서 보존 배열 모드를 써야 안전.
- **harmony 삽입 위치**: 적용 음표 **바로 앞**(measure music-data 시퀀스 내)에 넣어야 의미가 맞음. 엉뚱한 위치면 코드가 다른 박에 붙음.
- **lyric 삽입 위치**: `<note>` 내 `notations` **뒤, play 앞**. 앞쪽(pitch/duration 전후)에 넣으면 스키마 위반.
- **네임스페이스/DOCTYPE**: MusicXML은 기본 네임스페이스 미사용(XSD는 chameleon)이라 보통 namespace 없음. 단 **DOCTYPE(`<!DOCTYPE score-partwise PUBLIC ...>`)** 유무·xlink 접두사(`xmlns:xlink`)는 보존 필요. fast-xml-parser 기본은 DOCTYPE/PI/주석을 떨굴 수 있어 옵션(`processEntities`, `commentPropName`, DOCTYPE 수동 재삽입) 관리 필요.
- **속성/엔티티**: `default-x` 등 속성, `&`·악센트 문자 이스케이프, `xml:space`·`number` 속성 보존 주의. (근거: §2 note/harmony/lyric 순서 명세 및 일반 XML 직렬화 원리)

## 6. "음악적 sanity 체크" 아이디어

- **마디 박자 합 == time signature**: measure 내 음표·쉼표 duration 합(backup/forward 반영)이 `divisions × (beats × 4 / beat-type)`와 일치하는지. 불일치 = 누락/오삽입.
- **마디 수·파트 수 보존**: 편집 전후 `<part>` 개수, 각 part의 `<measure number>` 개수·번호 연속성 유지.
- **pitch 유효성**: step ∈ {A–G}, octave 정수 범위, alter ∈ {-2..+2}(미분음 제외).
- **harmony/lyric 정합**: harmony가 실제 음표 시점에 정렬되는지, lyric `number`(절)·syllabic 연결(begin/middle/end)이 음절 흐름과 맞는지.
- **divisions 일관성**: 동일 part 내 divisions 변경 시 duration 환산 깨짐 점검.
- **라운드트립 검증**: music21로 파싱→재내보내기 후 마디 수/음표 수/박자 합이 보존되는지 비교(가장 강력한 종합 sanity). (출처: §4의 music21 파서 검증)

# OMR 정확도 향상 가이드: 전처리·세그멘테이션·평가·대안 엔진 (출처 검증 완료)

## 1. 스캔/사진 악보 전처리

**DPI / 해상도**
- 최소 300 DPI 권장, 작은 기호가 많은 악보는 400 DPI, 실무 최적은 400~600 DPI. 핵심 기준: **두 보표선 사이 수직 간격이 약 20px**가 되도록 해상도 설정. 200 DPI 미만은 디테일 손실, 500 DPI 초과는 인식 이득 없이 CPU/메모리만 낭비. (Audiveris 공식 문서) https://audiveris.github.io/audiveris/_pages/guides/advanced/scanning/
- 종합 가이드도 300 DPI 이상 권장. https://five-lines.com/en/scan-and-digitize-sheet-music-the-complete-guide-2026/

**Grayscale / 이진화**
- **grayscale 입력을 흑백(1-bit)보다 선호** — Audiveris가 자체 adaptive 알고리즘으로 이진화하므로 스캐너 이진화보다 기호 왜곡이 적음. 즉 전처리에서 미리 1-bit로 굳히지 말고 grayscale을 넘겨라. (Audiveris) https://audiveris.github.io/audiveris/_pages/guides/advanced/scanning/
- **Otsu(전역 임계)**: 가장 빠르고 일반적으로 성공적이나, adaptive 대비 보표선 누락률(missed staff)이 높음. **Sauvola/Wolf(국소 적응형)**: 균일한 배경 노이즈 문서엔 Sauvola, 저대비(글자가 종이와 비슷하게 흐릿) 스캔엔 Wolf가 우수. (ResearchGate 음악 이진화 연구; ScanTailor 문서) https://www.researchgate.net/publication/221258807_Music_Score_Binarization_Based_on_Domain_Knowledge / https://scantailor.net/
- 효과: 적응형 이진화는 조명 불균일·복잡 레이아웃에서 정보 손실을 줄여 후속 보표/기호 검출 정확도를 높임.

**Deskew(기울기 보정) / 노이즈·배경 제거 도구 (실재 확인)**
- **ScanTailor Advanced** — 오픈소스. 국소 적응형 이진화(Sauvola/Wolf), deskew(수평 기울기 + 물리적 틸트/원근 왜곡 보정), 노이즈 제거, 페이지 분할. OCR 정확도를 70%→98%로 올린 사례 인용. https://scantailor.net/ / https://github.com/4lex4/scantailor-advanced
- **unpaper** — 오픈소스 후처리 도구. 스캔 시 생긴 어두운 가장자리/검은 영역 제거, 중심 오정렬·회전 자동 감지 후 deskew. https://github.com/unpaper/unpaper
- **OpenCV / ImageMagick** — deskew, grayscale 변환, 임계 처리, 형태학 연산에 범용 사용. ImageMagick으로 벡터/래스터 판별·해상도 확인도 가능. https://www.imagemagick.org/discourse-server/viewtopic.php?t=21963
- 주의: oemer 개발자는 deskew가 오히려 오류를 유발할 때 `--without-deskew` 사용을 권고 — 전처리 단계가 항상 +효과는 아님. https://github.com/BreezeWhite/oemer

## 2. 벡터 vs 래스터 PDF 판별

- **pdfimages** (poppler): `pdfimages -list file.pdf`로 임베드 이미지의 width/height/색공간/인코딩/DPI/크기 확인. **추출 결과가 0개면 순수 벡터** 확정. 대형 이미지가 300~600 DPI로 한 장씩 들어있으면 래스터(스캔/래스터화 인쇄). https://bitsgalore.org/2021/09/06/pdf-processing-and-analysis-with-open-source-tools.html
- **pdffonts** (poppler): 폰트 목록 출력. 텍스트가 텍스트로 복사되면(=벡터 글리프) 임베드 폰트로 식별 가능. 폰트가 없고 이미지만 있으면 스캔 PDF 신호. https://apryse.com/blog/development/raster-vector-text-what-is-inside-pdf
- **mutool** (MuPDF): `mutool show`로 내부 객체 타입 검사, `mutool extract`로 이미지·임베드 폰트 추출 → 페이지가 XObject 비트맵(/Width·/Height·/ColorSpace·/Filter)뿐인지, 벡터 경로/텍스트가 있는지 판별. https://bitsgalore.org/2021/09/06/pdf-processing-and-analysis-with-open-source-tools.html
- 보조 휴리스틱: 뷰어에서 400~800% 확대 시 픽셀화되면 래스터, 안 되면 벡터. https://www.nutrient.io/blog/vector-pdf/
- **순수 벡터 악보 경로**: 래스터 OMR을 거치지 말고 **PDFtoMusic Pro**로 직접 변환하는 것이 더 정확. 단 이 도구는 **표보 소프트웨어가 직접 생성한 "character" 형식 벡터 PDF에서만 동작하고 스캔 PDF엔 작동하지 않음**. 무료판은 MusicXML 내보내기 불가(Pro만 MusicXML 지원). https://www.myriad-online.com/en/docs/diffpro.htm / https://www.myriad-online.com/en/products/pdftomusicpro.htm

## 3. 보표/시스템 검출로 줄 단위 크롭

- **수평 투영 프로파일(horizontal projection profile)**: 가장 단순·직관적. 행별 흑색 픽셀 합의 피크가 보표선 위치 → 보표선 두께 추정·시스템 경계 산출에 사용. 단점: **이미지 스큐에 민감** → deskew를 먼저 해야 함. (그래서 §1 deskew가 선행 조건) https://www.researchgate.net/figure/Staff-lines-detection-and-removal-by-Horizontal-projection-a-Original-image-b_fig21_321118165
- **Hough 변환 + 수학적 형태학(morphology)**: 둘 다 전 보표선 검출 가능. Hough는 정확하나 런타임이 형태학보다 길다. 형태학(개폐 연산)은 빠르고 staff line 두께 추출·제거에 적합. https://ieeexplore.ieee.org/document/5631269/
- **Stable Paths / path-following**: 휘어진(곡선) 보표선·스큐에 강건한 대안. https://www.researchgate.net/publication/221050952_Staff_Line_Detection_and_Removal_with_Stable_Paths
- 좌표 산출 워크플로우: 투영/형태학으로 보표선 y좌표 클러스터 → 5선 묶음을 한 staff로, 인접 staff를 한 system으로 그룹화 → 각 system의 상/하단 y 경계(+여백)로 페이지를 줄 단위 크롭. 대부분 파이프라인은 "보표선 검출 → 제거 → 기호 분할·분류" 순서. https://www.semanticscholar.org/paper/Detecting-the-Staff-Lines-of-Musical-Score-with-and-Chen-Zhang/3a2cdd9f6244622e3575f58d5d7463f0383971ac
- 참고: homr는 보표선 제거 대신 **clef·bar line을 staff anchor로 삼고 보표선 간격(unit size)을 계산**해 원근 변형을 흡수하는 방식. https://github.com/liebharc/homr

## 5. 평가 지표 및 ground-truth 워크플로우 (오픈 구현 확인)

- **MV2H** (Multi-pitch, Voice, Meter, note Value, Harmony): 다성 음악 전사를 5개 음악학적 축으로 통합 정량 평가. **비정렬(non-aligned) 전사도 자동 정렬해 평가** → end-to-end 시스템·MusicXML ground truth에 적합. 키/박자 변화도 평가 가능. 오픈 구현: https://github.com/apmcleod/MV2H (논문 https://arxiv.org/pdf/1906.00566)
- **TEDn** (Tree Edit Distance with note flattening): MusicXML 트리에서 note 요소를 flatten 후 정규화 트리 편집거리(=gold 복원 거리로 나눔)로 점수화. 인간 평가와의 상관이 가장 좋다고 보고. 오픈 구현: `app.evaluation.TEDn` 단독 파이썬 파일(`zss`+`Levenshtein` 의존). https://github.com/ufal/olimpic-icdar24
- **Symbol Error Rate (SER)** 및 **OMR-NED**: SER은 완전 기호를 토큰으로 한 편집거리 기반 단일 점수. **OMR-NED = (I+D)/(N₁+N₂)** 로 SER을 확장해 음표머리·빔·음정·임시표·다이내믹·클레프·박자표·가사 등 **요소별 세분 오류 프로파일** 제공. 데이터셋 **Sheet Music Benchmark(SMB, 685페이지, **kern, Hugging Face 공개)**와 함께 발표. https://arxiv.org/html/2506.10488v1 / https://www.emergentmind.com/topics/optical-music-recognition-omr
- **Ground-truth 비교 워크플로우 (실무용 오픈 도구)**: SMB 논문이 개선한 **MusicDiff**(Greg Chapman 유지)로 파싱 불가한 OMR 출력까지 기호 비교 가능. 즉 (1) ground-truth를 MusicXML/**kern으로 확보 → (2) 엔진 출력과 MusicDiff/OMR-NED로 정렬·비교 → (3) MV2H/TEDn으로 보조 채점. https://arxiv.org/html/2506.10488v1
- 별도 오픈 OMR 벤치마크(Audiveris·homr·oemer를 SMB에서 OMR-NED로 비교)도 GitHub에 존재한다는 언급 확인. https://github.com/topics/oemer

# 구현 스택 베스트프랙티스 검증 (2026-06 기준)

## 1. Next.js 15 App Router — 대용량 파일 업로드 + 진행률 전달

- **Route Handler는 본문 크기 한계를 설정할 수 없음.** App Router의 Route Handler에는 Pages Router의 `api.bodyParser.sizeLimit` 같은 옵션이 없다. 본문 크기를 직접 제어하려면 Server Action(`serverActions.bodySizeLimit`)을 쓰거나, 업로드 자체를 Next 서버를 우회시켜야 한다. (https://github.com/vercel/next.js/issues/57501, https://nextjs.org/docs/messages/api-routes-response-size-limit)
- **권장 패턴: Next 서버로 바이트를 통과시키지 말 것.** 대용량 PDF는 클라이언트 → Supabase Storage 서명 URL로 직접 PUT 하고, Route Handler에는 메타데이터(파일 경로, job id)만 보낸다. 이러면 Next 본문 크기 한계, 메모리 버퍼링, 서버리스 페이로드 한계(과거 4MB 응답 제한 등) 문제를 전부 회피한다. (https://nextjs.org/docs/messages/api-routes-response-size-limit)
- **proxy 사용 시 버퍼 한계 주의(Next 15.5+/16).** `experimental.proxyClientMaxBodySize` 기본값은 **10MB**다. 한계를 넘으면 에러가 아니라 **본문이 한계까지만 버퍼링되고 경고만 로깅**되며 요청은 부분 본문으로 계속 진행된다(조용한 데이터 손상 위험). 아직 experimental이며 단위는 `b/kb/mb/gb`. (https://nextjs.org/docs/app/api-reference/config/next-config-js/proxyClientMaxBodySize)
- **알려진 버그:** Next 15에서 FormData로 대용량 파일 업로드 시 요청 본문의 바이너리가 드롭되는 사례 보고가 있음 — FormData 멀티파트보다 서명 URL 직접 업로드가 더 안전한 또 하나의 근거. (https://github.com/vercel/next.js/discussions/86985)
- **진행률 전달 — 폴링 vs SSE:**
  - SSE는 App Router에서 Route Handler + `ReadableStream`으로 깔끔하게 구현되며 단방향 server→client push에 적합(진행률 바, 상태 업데이트의 전형적 사용처). 폴링 대비 지연·서버 부하가 낮다. (https://xiouyang.medium.com/building-production-ready-sse-in-next-js-a-complete-guide-18450fb74b7a, https://dev.to/richardlau/implementing-real-time-status-updates-with-server-sent-events-in-nextjs-4da1)
  - **현실적 절충:** OMR 잡은 수십 초~분 단위이고 워커(별도 프로세스)가 상태 소유자다. SSE 스트림을 살아 있게 유지하려면 서버리스/엣지에서는 함수 실행 시간 제한과 충돌하고, 워커→웹 간 진행률 전달용 Redis pub/sub 브릿지가 추가로 필요하다. 저빈도(2~5초) 폴링이 인프라가 단순하고(상태 무관 GET으로 jobs 테이블 조회) 서버리스 친화적이라, MVP에서는 폴링이 합리적 기본값이고 SSE는 체감 지연을 줄이려는 후속 최적화로 보는 게 타당하다. (https://dev.to/haraf/server-sent-events-sse-vs-websockets-vs-long-polling-whats-best-in-2025-5ep8)

## 2. BullMQ + Redis 워커 패턴 (CPU 무거운 OMR)

- **CPU 바운드에는 높은 concurrency가 역효과.** concurrency는 Node 이벤트 루프를 활용하므로 **IO 무거운** 잡에만 처리량 이득이 있다. CPU 집약 잡은 concurrency를 올려봐야 오버헤드만 늘고 처리량이 떨어진다. → OMR은 낮은 concurrency(예: 1~2)로 두는 게 정석. (https://docs.bullmq.io/guide/workers/concurrency)
- **샌드박스 프로세서(Sandboxed processors)를 쓸 것.** CPU 무거운 처리는 이벤트 루프를 점유해 BullMQ의 잡 락 연장 등 부킹이 막혀 잡이 **"stalled"** 로 빠진다. 샌드박스 프로세서는 처리 로직을 별도 프로세스에서 돌려 부킹 코드와 분리하므로 stall이 잘 안 생긴다. (https://docs.bullmq.io/guide/workers/sandboxed-processors)
- **진짜 병렬성은 "워커를 여러 개" 띄워서 얻는다.** 같은 Redis 큐를 소비하는 워커 인스턴스를 여러 머신/컨테이너에 띄우는 게 BullMQ 권장 스케일 방식이며 가용성도 함께 확보된다. 단일 워커의 concurrency를 키우는 것보다 권장됨. (https://docs.bullmq.io/guide/parallelism-and-concurrency, https://github.com/OptimalBits/bull/discussions/2098)
- **실패 재시도:** 워커가 아니라 잡 추가 시점에 `attempts` + `backoff`(예: exponential)로 지정. OMR처럼 외부 프로세스(Audiveris)가 비결정적으로 실패할 수 있으므로 재시도 한도와 백오프를 명시하고, 멱등성(같은 입력 재처리 안전)을 유지할 것. (https://docs.bullmq.io/guide/workers, https://dev.to/asad_ahmed_5592ac0a7d0258/building-scalable-background-jobs-in-nodejs-with-bullmq-a-complete-guide-509p)
- **Audiveris를 자식 프로세스로 spawn하는 경우** 잡 락이 stall될 위험이 줄지만(별도 OS 프로세스), 그래도 `lockDuration`을 OMR 최대 실행 시간보다 충분히 길게 잡고, 동시 실행 OMR 수만큼 CPU/메모리를 확보할 것.

## 3. Supabase Storage + Postgres jobs 테이블

- **service_role(시크릿) 키는 서버 전용.** RLS를 우회하는 JWT이므로 워커/큐 프로세서/Edge Function 같은 서버 측 컴포넌트에서만 사용하고 절대 클라이언트에 노출 금지. 정기 잡·큐 프로세서가 정확히 시크릿 키의 정당한 사용처로 명시됨. (https://supabase.com/docs/guides/getting-started/api-keys, https://supabase.com/docs/guides/troubleshooting/performing-administration-tasks-on-the-server-side-with-the-servicerole-secret-BYM4Fa)
- **업로드 패턴:** `createSignedUploadUrl`로 서버가 토큰을 발급 → 클라이언트가 `uploadToSignedUrl`로 직접 업로드. 이걸로 1번 항목의 Next 본문 크기 문제를 우회한다. (https://supabase.com/docs/reference/javascript/storage-from-createsigneduploadurl, https://supabase.com/docs/reference/javascript/storage-from-uploadtosignedurl)
- **다운로드/결과 제공:** 결과 .mxl은 비공개 버킷에 두고 `createSignedUrl`(만료 시간 지정)로 제공. 서명 URL은 Auth JWT와 분리된 내부 키로 서명되어 Auth 키 변경과 무관하게 만료까지 유효. (https://supabase.com/docs/guides/storage/serving/downloads, https://supabase.com/docs/guides/storage/buckets/fundamentals)
- **jobs 테이블:** Supabase 공식의 "jobs 테이블" 전용 패턴 문서는 없으나, 일반 패턴은 Postgres에 `jobs(id, user_id, status, input_path, output_path, progress, error, created_at)`를 두고 status를 워커가 service_role로 갱신, 클라이언트는 RLS(`user_id = auth.uid()`)로 자기 잡만 조회. (Storage↔DB 메타데이터 연동은 표준 패턴) (https://dev.to/kanta13jp1/supabase-storage-deep-dive-bucket-design-signed-urls-image-transforms-and-rls-3b9k)

## 4. OSMD를 Next.js에서 로드 + .mxl 직접 로드

- **동적 import(`ssr:false`) 필수.** OSMD는 브라우저 DOM/VexFlow 의존이므로 SSR을 끄고 클라이언트에서만 마운트해야 한다. `next/dynamic`의 `{ ssr: false }`로 로드하고 로딩 표시를 두는 것이 권장 패턴. (https://opensheetmusicdisplay.github.io/usage/typescript/, https://github.com/opensheetmusicdisplay/opensheetmusicdisplay/wiki/Getting-Started)
- **`.mxl` 직접 로드 가능 — 별도 zip 해제 불필요.** `osmd.load()`는 URL 문자열, 원시 XML 문자열, DOM Document, **Blob**을 받으며 **압축된 MXL을 직접 처리**한다(내부적으로 압축 해제). 즉 파일 input/드래그앤드롭의 File(=Blob)을 그대로 `osmd.load(file, file.name)`에 넘기면 된다. 앱에서 수동으로 JSZip 등으로 unzip할 필요 없음. (context7 `/opensheetmusicdisplay/opensheetmusicdisplay` `osmd.load` 문서, https://opensheetmusicdisplay.github.io/classdoc/classes/OpenSheetMusicDisplay.html)
  - 주의: Storage의 서명 URL을 그대로 `load(url)`에 넘기면 OSMD가 fetch하므로 CORS 설정이 필요할 수 있다. 안전하게는 앱이 fetch해서 Blob으로 넘기는 방식 권장.

## 5. docker-compose: web / worker / redis + Audiveris

- **표준 구성:** `redis`(healthcheck 포함) + `web`(Next standalone) + `worker`(BullMQ). 워커는 `depends_on: redis (condition: service_healthy)`, 환경변수로 Redis 접속/큐명/concurrency 주입. (https://oneuptime.com/blog/post/2026-01-21-bullmq-workers-docker/view, https://oneuptime.com/blog/post/2026-02-08-how-to-set-up-a-nextjs-postgresql-redis-stack-with-docker-compose/view)
- **Next는 standalone 출력 + 멀티스테이지 빌드.** `output: 'standalone'`로 빌드해 최종 이미지에 전체 node_modules가 아닌 산출물만 담아 이미지 크기를 줄인다. (https://dev.to/whoffagents/docker-compose-for-full-stack-development-nextjs-postgres-redis-and-production-builds-57m8)
- **Audiveris 결합 시 핵심 주의:**
  - Audiveris는 **Java 기반 OMR**이라 워커 이미지에 **JRE/JDK + Audiveris 바이너리 + 폰트/네이티브 라이브러리**가 필요하다. Node 워커 이미지와 Java 런타임을 한 이미지에 합치면 이미지가 무거워지므로, **Audiveris를 별도 이미지/서비스로 분리**하고 Node 워커가 `docker exec`/HTTP/공유 볼륨으로 호출하거나, Node 워커 이미지에 Java를 멀티스테이지로 얹는 두 가지 선택지가 있다. (https://github.com/weidi/audiveris-docker, https://z-uo.medium.com/audiveris-on-docker-container-8c642c75e2a7)
  - **배치/헤드리스 호출:** `Audiveris -batch -export <input>` 형태. `-batch`(GUI 없음), `-export`(MusicXML/.mxl 출력). 압축 출력 여부는 `-option org.audiveris.omr.sheet.BookManager.useCompression=true/false`로 제어(true면 .mxl). (https://audiveris.github.io/audiveris/_pages/guides/advanced/cli/, https://github.com/Audiveris/audiveris/wiki/CLI-Arguments)
  - **볼륨 공유:** 입력 PDF와 출력 .mxl을 web/worker/Audiveris가 공유 볼륨(또는 Storage 다운로드 후 로컬 tmp)으로 주고받게 설계. 컨테이너 예: `docker run --rm -v ./:/data .../Audiveris -batch -export ...`. (https://github.com/weidi/audiveris-docker)
  - **리소스:** OMR은 CPU·메모리(JVM 힙) 집약 → 워커/Audiveris 컨테이너에 CPU·메모리 한도와 concurrency를 함께 묶어 한 호스트에서 OMR 동시 실행 수를 통제할 것(2번 항목과 연계).

## 6. Headless OSMD 스모크 테스트 (Node 렌더 현실성)

- **현실적이고 공식 지원됨.** OSMD는 "browsers and headless with NodeJS"로 렌더한다고 README에 명시. 저장소에 동봉된 `test/Util/generateImages_browserless.mjs` 스크립트로 브라우저 없이 Node에서 MusicXML/MXL을 SVG/PNG로 출력한다. (https://github.com/opensheetmusicdisplay/opensheetmusicdisplay, https://github.com/opensheetmusicdisplay/opensheetmusicdisplay/wiki/Exporting-PNG,-SVG-and-PDF)
- **시그니처:** `node generateImages_browserless.mjs <osmdBuildDir> <sampleDir> <imageDir> <svg|png> [width|0] [height|0] [filterRegex|all|allSmall] [--debug|--osmdtesting]`. width=0 height=0이면 단일 페이지(endless) 렌더. (context7 `/opensheetmusicdisplay/opensheetmusicdisplay` 문서)
- **스모크 테스트로서의 의미:** 변환 산출 .mxl을 헤드리스 Node에서 OSMD가 **에러 없이 파싱·렌더**되는지 확인하면, 결과가 OSMD 프론트엔드에서도 깨지지 않음을 CI에서 검증 가능 → 실용적인 회귀 게이트. 다만 내부적으로 JSDOM 등 DOM 셋업과 OSMD 빌드 산출물(`build/`)이 필요하므로, 단순 `import` 한 줄이 아니라 이 공식 스크립트 경로를 따르는 것이 안정적이다. (https://github.com/opensheetmusicdisplay/opensheetmusicdisplay/wiki/Testing)

## 종합 권고 (검증된 핵심)
- 업로드 바이트는 Next를 거치지 말고 **Supabase 서명 업로드 URL로 직접** → Route Handler 본문 한계/버퍼링 버그 회피.
- OMR 워커는 **샌드박스 프로세서 + 낮은 concurrency(1~2) + 워커 인스턴스 수평 확장**, 잡에 `attempts`+exponential `backoff`.
- **service_role 키는 워커 전용**, jobs 테이블 status는 워커가 갱신, 클라이언트는 RLS로 자기 잡 조회 + 폴링(또는 후속 SSE).
- OSMD는 **`ssr:false` 동적 import**, **.mxl Blob 직접 `load()`**(수동 unzip 불필요).
- Audiveris(Java)는 **Node 워커와 이미지 분리** 또는 멀티스테이지로 Java 동봉, `-batch -export` + `useCompression` 옵션, 공유 볼륨.
- 헤드리스 OSMD 스모크 테스트는 **공식 `generateImages_browserless.mjs`** 경로로 CI 회귀 게이트화.

(주의: `proxyClientMaxBodySize`는 아직 experimental이며, Audiveris의 정확한 Docker 이미지 태그/버전은 위 커뮤니티 이미지 기준이므로 채택 전 최신 릴리스 확인 권장.)
